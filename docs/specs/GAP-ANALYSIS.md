## 3. 差异项详细分析

### 3.1 高优先级差异（P0 - 必须解决）

#### 3.1.1 碳核算模型未完整实现

**原始需求**：
- 发电企业碳核算模型（25个原始数据输入参数）
- 电网企业碳核算模型（9个原始数据输入参数）
- 基于国家标准《中国发电企业温室气体排放核算方法与报告指南》

**当前实现**：
- CarbonReport 实体存在，但未集成精确计算公式
- 缺少原始数据输入字段的完整定义

**影响**：
- 碳核算结果准确性无法保证
- 无法满足原始需求的核心业务要求

**建议**：
1. 扩展 CarbonReport 实体，增加原始数据输入字段
2. 实现 `CarbonCalculationService`，集成原始公式
3. 支持发电企业和电网企业两种模型

#### 3.1.2 Fabric区块链未真实对接

**原始需求**：
- 基于 Hyperledger Fabric 的区块链底层
- 企业节点、审核员节点、管理员节点、监管机构节点
- 智能合约执行碳交易

**当前实现**：
- BlockchainService 为模拟实现
- 无真实 Fabric 网络对接

**影响**：
- 数据不可篡改性依赖传统数据库
- 无法体现区块链核心价值

**建议**：
1. 短期：保持模拟模式，作为可演示功能
2. 中期：对接 Fabric 测试网络
3. 长期：部署生产级 Fabric 网络

### 3.2 中优先级差异（P1 - 应该解决）

#### 3.2.1 AI预测模块为Stub实现

**原始需求**：
- AI接口动态碳排放预测
- 预测结果以可视化图表呈现
- 预测可信度评分

**当前实现**：
- CarbonPredictionService 使用简单线性回归
- 标注为 Stub 实现

**影响**：
- 预测准确性有限
- 无法提供真正的智能建议

**建议**：
1. 接入专业ML模型（如Prophet、LSTM）
2. 增加企业境况智能推断功能
3. 优化预测可信度算法

#### 3.2.2 身份认证未采用区块链方案

**原始需求**：
- 基于区块链的身份认证算法
- 用户身份与节点类型匹配

**当前实现**：
- 传统 JWT 认证
- 基于角色的访问控制

**影响**：
- 无法实现去中心化身份
- 依赖传统认证体系

**建议**：
1. 评估区块链身份认证的必要性
2. 如必要，集成 Fabric CA

### 3.3 低优先级差异（P2 - 可选解决）

#### 3.3.1 企业境况智能推断未实现

**原始需求**：
- 基于企业账户历史活动、信誉分、数据、企业客观信息
- 推断企业的发展状况和潜在风险
- 推测是否违规、虚报排放

**当前实现**：
- 未明确实现

**建议**：
1. 作为 AI 模块扩展功能
2. 集成到 CreditScoreService 或新增服务

**第一阶段（P0 - 2周）：**
1. 实现发电企业碳核算模型（25参数）
2. 实现电网企业碳核算模型（9参数）
3. 扩展 CarbonReport 实体字段

1. 接入专业ML预测模型
2. 实现企业境况智能推断
3. 完成Phase 4碳中和项目测试

**第三阶段（P2 - 按需）：**
1. Fabric区块链真实对接
2. 区块链身份认证评估
